Troubleshooting 01 — CGMS

BLE 백필링 큐 설계 — 재연결 시 누락 데이터를 자동 복구하는 체이닝 구조

재연결 몇 번이든 요청은 1회 · 끊긴 체인은 다음 실시간 데이터가 자동 복구
문제
  • 센서 재연결 시 놓친 구간을 다시 요청하는 백필링이, 재연결마다 중복 실행됐다. 어떤 요청이 나갔는지 기록도 없었다.
해결
  • 누락 구간을 5분→1분→1초 순으로 큐에 적재. 1초 단위는 왕복 지연이 커서 누락 폭이 작을 때만 요청.
  • invokeNext()가 큐에서 하나 꺼내 처리하고 자신을 다시 부른다. 상태 enum 없이 큐가 비면 끝, 남으면 진행 중.
  • 응답이 끊기면 다음 실시간 insert가 trigger()로 멈춘 체인을 되살린다.

명시적 상태 enum 없이 큐의 존재/부재만으로 진행을 판단하는 self-driving 체인.

결과
  • 재연결 몇 번이든 요청은 1회. 중복은 uniqueKey로 큐 진입 시점에 차단.
  • 응답이 유실돼도 다음 실시간 데이터가 도착하는 순간 체인이 스스로 복구.
Troubleshooting 02 — CGMS

앱 생명주기 추적 — Cold Start 시 BLE 초기화가 누락되던 .task 타이밍 버그 해결

Cold Start BLE 연결 누락 해결 · Launch 5종 판정 · Apple DTS 직접 협의로 규명
문제
  • 앱 최초 실행 시 CoreBluetooth 초기화를 SwiftUI View.task에 걸어뒀는데, .task는 뷰가 떠야 실행된다. 뷰 없이 프로세스만 깨어나는 경로(BLE State Restoration 등)에선 호출되지 않아 BLE 연결이 간헐적으로 누락됐다.
  • Jetsam 종료는 applicationWillTerminate가 안 불려 로그조차 없었다.
해결
  • BLE 초기화를 View.task → AppDelegate.didFinishLaunching으로 이전. 뷰 타이밍과 무관하게 launch마다 1회 보장. State Restoration은 더 이른 willFinishLaunching에서 선제 처리.
  • ActivePrewarm·applicationState·launchOptions 조합으로 Launch 유형 5종(Cold / Background Cold / Background / Pre-warm / Warm Resume) 판정.
  • Apple DTS 엔지니어와 직접 협의(Forums Thread 763739·765674)해 "진짜 Pre-warm은 didFinishLaunching까지 오지 않는다"를 확정 — Pre-warm 오판을 바로잡음.
  • cleanExitFlag로 정상 종료 vs Jetsam 구분, launch마다 dead duration·발열 원격 수집.
결과
  • 모든 launch 경로에서 BLE 초기화 1회 보장. 재발 시 어느 경로에서 끊겼는지 로그로 추적 가능.
Troubleshooting 03 — CGMS

로컬 알림 시스템 설계 — 재부팅·삭제·언어 변경에도 깨지지 않는 일관성

재부팅·타임존·언어 변경에도 알림 일관성 유지 · 삭제 후에도 재알림 판정 정상
문제
  • 재부팅·앱 재시작 시 OS 알림 상태와 DB가 어긋났다. 알림함에서 항목을 지우면 재알림 간격 판정용 레코드까지 사라졌고, 언어를 바꾸면 예약 알림이 이전 언어로 떴다.
해결
  • 앱 시작 시 OS 알림과 DB를 대조해 불일치 자동 복구. 빌드·타임존 변경 감지 시 전체 재등록.
  • 삭제는 Soft Delete — 판정용 레코드는 DB에 남긴다. 레코드가 누적되는 비용은 의료기기 특성상 판정 정확도를 우선해 감수.
  • 알림 문자열은 등록 시점이 아닌 표시 시점에 평가(localizedUserNotificationString) — 언어를 바꿔도 현재 언어로 표시.
결과
  • 재부팅·재시작·타임존·언어 변경 어떤 상황에서도 알림과 DB가 어긋나지 않는다.

혈당 타임라인 차트 재설계 — 슬롯 인덱스에서 Date 기반 Viewport로

프레임 드랍 828회→1회 · SwiftUI 소요 33.7s→2.8s · 60fps 스크롤

Architecture — CGMS

모듈화 아키텍처 설계 — DIP로 의존을 끊어 센서 없이 검증 가능한 구조로

Feature = 4타겟 마이크로 모듈 · 주입만 바꿔 센서 없이 단독 실행 · 5개 레이어 52개 모듈
문제
  • 단일 타겟이라 화면 하나 고쳐도 전체 재빌드. BLE 센서를 실제로 부착·웜업하지 않으면 차트 UI를 개발·검증할 수 없었고, 저혈당·고혈당 같은 경계 시나리오는 실제 혈당으로 재현이 불가능했다.
해결 — Feature 마이크로 모듈화
  • Feature 하나를 Sources·Testing·DemoApp·Tests 4타겟 마이크로 모듈로 구성. 직접 만든 Tuist 템플릿이 이 구조를 강제해서, 누가 모듈을 추가해도 같은 모양이 나온다.
  • Sources(화면·로직)는 Domain의 Repository 프로토콜에만 의존한다(DIP). BLE·SwiftData 실구현은 Data 모듈이, Mock·샘플 데이터는 Testing 타겟이 소유.
  • 그래서 같은 화면 코드가 주입만 바꿔 두 곳에서 돈다 — 본 App은 실구현을 꽂아 실센서로, DemoApp은 Testing의 Mock을 꽂아 센서 없이 시뮬레이터에서 단독 실행.
  • DemoApp의 Hypo/Hyper 슬라이더로 저혈당·고혈당 경계 시나리오를 즉시 재현. 전체 구조는 5개 레이어 52개 모듈.

화면 코드(Sources)는 하나 — Domain 프로토콜 뒤의 구현만 바꿔 끼우면 본 App은 실센서로, DemoApp은 더미로 같은 화면이 돈다.

DemoApp의 Hypo/Hyper 슬라이더. 센서 없이 경계값을 조작해 세 차트의 색 구간 전 시나리오를 검증한다.

설계 판단
  • TCA UI Feature는 Interface로 분리하지 않았다 — Parent가 Child State를 직접 소유해야 Scope가 동작하기 때문. 추상화는 Domain/Infra 레이어에만.
  • liveValue는 구현을 소유한 Data 모듈에, Interface에는 testValue만 — 구현 변경이 상위 레이어로 전파되지 않게.
  • 샘플 데이터는 Domain/Testing(모델 헬퍼)과 Feature/Testing(화면 시나리오)으로 분리 — Domain이 화면 사정을 모르게.
결과
  • 센서 부착·웜업 없이 차트 전 시나리오를 검증할 수 있게 됐다.

멀티에이전트 역할 분배 + Plan-first 워크플로우

Claude / Gemini / Codex 역할 분리 · 파일 충돌 방지 · AI 반복 실수 차단 시스템
문제
  • AI에게 바로 구현을 시키면 사이드이펙트를 놓쳐 같은 실수를 반복하거나 이전 작업을 되돌렸다. 단일 에이전트로 분석·파싱·구현을 다 시키면 컨텍스트가 오염되고 토큰 비용이 커졌다.
설계
  • Plan-first — 구현 전 계획 제시·승인을 강제. 멀티스텝 작업은 단계마다 검증 방법을 붙여 계획한 뒤 실행.
  • 역할 분배 — Claude는 계획·판단, Gemini는 규격서 파싱·대용량 분석(1M 컨텍스트), Codex는 구현·반복 작업. 병렬 실행의 파일 충돌은 AGENTS.md의 파일 lock 규칙으로 해결.
  • 반복 실수는 known-gotchas.md에 기록해 다음 세션에서 자동 차단.
결과
  • AI가 모듈 의존 방향·네이밍 규칙·금지 패턴을 처음부터 준수. revert 반복 없이 계획 → 승인 → 구현 흐름 유지, 병렬 실행 파일 충돌 0건.
